# AI海龟汤游戏 PRD
> 版本：v1（MVP规划）
> 更新时间：2026-03-19

## 1. 产品概述
AI驱动的海龟汤游戏：玩家阅读“汤面故事”，向“AI主持人”提问；主持人只用 **`是` / `否` / `无关`** 三类回答，帮助玩家逐步推理并最终查看“汤底真相”。

## 2. 目标与成功标准
### 2.1 目标
1. 用“只回答三类”的对话机制，让玩家可以顺畅完成一局推理。
2. 让用户体验具备“悬疑、沉浸、揭晓仪式感”的视觉与交互。
3. 控制 AI 输出格式稳定，避免模型跑题/输出多余文本。

### 2.2 成功标准（MVP可落地指标）
1. 单局可完成率：进入游戏并查看汤底的用户占比（建议 KPI：>= 60%）。
2. 平均提问轮数：在可接受成本内完成推理（建议 KPI：<= 20 轮）。
3. AI回答格式正确率：返回 `是/否/无关` 的比例（建议 KPI：>= 98%）。
4. 关键链路体验：`提问->回答` 平均耗时与失败率（建议目标：P95 < 2s；错误率 < 1%）。

## 3. 用户与使用场景
### 3.1 目标用户
1. 喜欢海龟汤、文字推理、悬疑内容的用户。
2. 想快速开局、不需要复杂学习成本的用户。

### 3.2 核心场景
用户在大厅选择一个海龟汤 -> 在游戏页面反复提问 -> 查看汤底 -> 可再开一局。

## 4. 需求范围（MVP）
### 4.1 MVP必须做的
#### 4.1.1 游戏大厅
- 展示游戏列表（至少包含：标题、难度）。
- 点击进入：创建新的“游戏会话”（与某个汤面故事绑定）。

#### 4.1.2 游戏页面（对话推理）
- 展示汤面故事内容。
- 提供输入框，允许玩家输入问题并提交。
- AI只允许回答三类之一：`是` / `否` / `无关`（禁止额外解释）。
- 展示历史对话（按时间顺序）。
- 提供按钮：
  - `查看汤底`：进入汤底页面。
  - `结束游戏`：结束会话并返回（大厅或保持当前页面的结束态，以实现为准）。

#### 4.1.3 汤底页面（揭晓）
- 展示完整汤底（真相）。
- 展示玩家推理过程（至少包含玩家提问 + AI回答的时间线）。
- `再来一局`：返回大厅或重新开始同一故事（由实现选择，但体验要一致）。

### 4.2 后续可以做的（不在MVP）
- 用户登录系统
- 游戏记录保存
- 排行榜
- 自定义海龟汤
- 多人对战模式

## 5. AI交互与输出约束（关键）
### 5.1 输出格式要求
- AI必须严格输出以下之一（区分大小写/空格以实现为准，建议做服务端解析）：
  - `是`
  - `否`
  - `无关`

### 5.2 提示词与解析策略（建议实现思路）
- 在请求AI时附带“系统指令”，明确：只能输出三类之一，且不能输出解释、理由、标点前缀等。
- 服务端对模型输出做校验/解析：
  - 若输出不合法：触发重试或回退为 `无关`（策略在实现中确定）。

### 5.3 业务规则（建议待确认）
- `无关` 的语义：当问题无法判断/超出汤面信息/需要额外背景时如何判定（建议统一定义）。
- 是否允许玩家重复或变相提问：是否做限流/频控（MVP可先不做或做简单限流）。

## 6. 界面与交互要求
- 整体风格：神秘悬疑、深蓝色调。
- 游戏大厅：卡片式布局，显示故事标题和难度。
- 游戏页面：聊天界面风格，类似微信（气泡/时间线/加载态）。
- 汤底页面：揭晓真相的仪式感（可用动画/渐显/强调文案）。

## 7. 技术栈建议（保持可替换）
- 前端：React + TypeScript + Tailwind CSS
- 后端：Node.js + Express
- AI：deepseek
- 部署：Vercel

## 8. 非功能性要求（MVP）
- 稳定性：AI接口失败要有可理解的错误提示与重试/回退策略。
- 性能：提问链路尽量控制响应时间（见成功标准）。
- 成本：限制单局最大轮数（或对话长度）以避免费用失控。
- 安全与内容：限制输出为三类，减少不必要的“长文本/敏感信息”风险（仍需视模型策略进一步完善）。

## 9. 风险与待决问题
1. 模型可能输出多余内容或偏离格式：需要服务端校验与兜底策略。
2. 汤面/汤底故事数据格式与组织方式：当前PRD未定义，可能影响落地速度。
3. “难度”的定义与展示逻辑：难度是人为标注还是由故事复杂度推断（建议提前定一下）。

## 10. 验收清单（MVP）
- 能在大厅看到至少1个游戏/故事并开始一局。
- 游戏页面可进行多轮提问，AI始终只返回 `是/否/无关`。
- 能点击“查看汤底”并在汤底页面看到真相与对话时间线。
- “再来一局”可正常开启下一局（不会串会话历史）。
